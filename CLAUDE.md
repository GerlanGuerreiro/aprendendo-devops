## 🎯 Filosofia do Projeto
O objetivo real desta pasta é **a trilha de aprendizado DevOps em si** — não uma aplicação específica. Cada módulo pratica um conceito de infraestrutura usando um ou mais "labs" (projetos-cobaia) dentro da pasta `labs/`. O primeiro lab é uma **API de Telemetria/Clima (FastAPI + PostgreSQL)**, mas ela é só o payload usado pra exercitar cada camada (venv, containers, orquestração, CI/CD) — não o produto final.

**Princípio de host limpo:** instalar o mínimo possível diretamente na máquina do Gerlan. A partir do Módulo 2 (VM/KVM), o trabalho prático migra para dentro de uma VM isolada acessada via SSH — como uma protoboard separada da placa de produção, onde erro de instalação não polui o ambiente principal. O host só recebe ferramentas de controle remoto (Git, cliente SSH) e o hypervisor (custo de entrada único para ganhar isolamento).

**Proficiência em Claude Code é objetivo paralelo da trilha**, não só ferramenta de apoio. A prática acontece de forma gradual em todo módulo (plan mode, subagents, fluxo de git via Claude Code), e é consolidada formalmente no Módulo 11, quando já existe infraestrutura real (VM, containers, IaC) para automatizar de verdade.

Rede confirmada com `ping -c 3 8.8.8.8` (0% perda) através de NAT em dupla camada: `lxcbr0` (`MASQUERADE` que o `lxc-net` já configura sozinho, mesma regra que foi escrita manualmente no Módulo 4 Estágio 2) → `virbr0` (NAT do Módulo 2) → internet. `cgroups` testado com `lxc-cgroup`: baseline de memória do container em ~18,9MB, teto artificial de `memory.max` setado em 50MB, depois estourado de propósito de dentro do container (`perl -e '$x = "a" x (150*1024*1024); sleep 10;'`) — resultado didático melhor que o esperado: o processo **não** foi morto, e `memory.events` mostrou `max: 1882` (o teto foi atingido e forçou reclamação de memória, provavelmente via swap da VM) mas `oom_kill: 0` — achado que corrigiu uma simplificação inicial da explicação (`memory.max` é um teto que aciona reclamação primeiro, `oom_kill` só acontece se a reclamação falhar, não é "mata na hora"). Teste de forçar o `oom_kill` de verdade (via `memory.swap.max 0`) fica como próximo passo opcional, não executado por falta de tempo na sessão.
  - Checkpoint de entendimento validado ao longo do módulo (comparação chroot vs. namespace vs. LXC vs. VM, e a implicação de segurança de PID/UID compartilhados no `chroot`) — Gerlan respondeu corretamente sem precisar de correção de conceito nesta sessão.
- [x] **Módulo 6: Conteinerização Moderna (Docker & Podman)** `✅ Concluído`
  - Criação de Dockerfile profissional, otimização de camadas.
  - Migração para Podman (entendimento do conceito Rootless e segurança corporativa).
  - **✅ 2026-07-22: módulo concluído em 4 estágios, todos com prova prática dentro da VM.**
  - **Estágio 1 (instalação):** Docker CE instalado via repositório oficial (não o `docker.io` do Debian) — motivo: traz `docker-buildx-plugin` e `docker-compose-plugin` (Compose v2) já integrados, necessários pro Módulo 7; `docker.io` do Debian vem mais antigo e sem esses plugins. `docker run hello-world` confirmado de ponta a ponta. Achado real: usuário `gerlan` inicialmente sem grupo `docker`, todo comando pedia `sudo` — corrigido com `usermod -aG docker gerlan` + sessão SSH nova (grupo só aplica em login novo). Discussão de segurança aprofundada a partir disso: grupo `docker` equivale a acesso root no host, porque quem "aperta o gatilho" de qualquer comando é o `dockerd` (daemon que roda como root), não o usuário — não é sandbox parcial, é uma porta lateral pra root, o mesmo princípio do `chroot` sem isolamento de UID visto no Módulo 5, só que embrulhado num daemon. Aceito conscientemente pra ambiente de laboratório; motivo real de existir do Podman rootless (Estágio 4) registrado a partir daqui.
  - **Estágio 2 (Dockerfile + otimização de camadas):** `Dockerfile` da `telemetria-api` criado com `python:3.13-slim`, camada de dependências (`COPY requirements.txt` + `RUN pip install`) separada da camada de código (`COPY app/`), e usuário não-root (`useradd`/`USER appuser`) — defesa em profundidade independente da questão do grupo `docker` do host. `.dockerignore` criado (`venv/`, `__pycache__/`, `.git/`, `scripts/`) pra manter o contexto de build enxuto. Otimização de camadas **provada na prática, não só declarada**: rebuild após mudança de código (`v1`→`v2`) mostrou `CACHED` nas camadas de dependência e rebuild só na de código — mas revelou que `RUN useradd`, posicionado depois do `COPY app/`, também era invalidado em cascata (regra do Docker: uma camada mudar invalida **todas** as seguintes, mesmo sem relação de conteúdo). Corrigido subindo `useradd` pra antes do `COPY app/` (`v3`) — confirmado no rebuild seguinte que `useradd` passou a vir `CACHED` e só a camada de código invalidou.
  - **Estágio 3 (build + run):** container rodando e testado (`docker run -d --name telemetria-api -p 8000:8000`), `/health` respondendo, log JSON de startup íntegro. Conexão com o Módulo 5: `Started server process [1]` no log confirma PID 1 **dentro** do container, PID diferente (maior) confirmado no host via `docker top` — mesmo mecanismo de namespace de PID do LXC, agora com imagem em camadas por baixo em vez de rootfs solto. Discussão sobre build vs. run: `docker build` nunca toca em containers existentes (são operações totalmente independentes — build só produz imagem); trocar a versão de um container rodando exige parar+remover+subir de novo manualmente (`docker stop`+`docker rm`+`docker run` com a tag nova) — não existe atualização automática no Docker puro, ponto que antecipa a motivação real do Compose/K3s no Módulo 7. Esclarecidas também duas confusões conceituais: `--name` (nome do container, decidido no `docker run`) não tem relação com a tag da imagem (`-t`, decidida no `docker build`) — são dois namespaces completamente separados, só coincidiram no nome escolhido; e tags de imagem (`:v1`/`:v2`/`:v3`) funcionam como tags do Git (ponteiros pra um build imutável específico, não sobrescrevem histórico).
  - **Estágio 4 (Podman rootless):** Podman 5.4.2 instalado, mesmo Dockerfile buildado sem alteração (compatibilidade via padrão aberto OCI). Container rodando rootless, sem `sudo` em nenhum momento (`podman run`, porta 8001 pra não colidir com o container Docker). Diferença estrutural explicada e **provada com dados de kernel**: Docker usa daemon central (`dockerd`) sempre ativo como root; Podman não tem daemon (`conmon` supervisiona por container, rodando como o próprio usuário). Prova em 3 camadas: (1) `ps aux` no host mostrou o processo do container Docker como `gerlan` (coincidência de UID numérico entre `appuser` e o usuário do host, não mecanismo de segurança real) vs. o do Podman como `100999` (UID inexistente em `/etc/passwd`, vindo da faixa `/etc/subuid`); (2) forçado `--user root` num container Podman, `whoami` de dentro confirmou `root`, mas o host viu esse mesmo processo como `gerlan` — root de dentro mapeado pro próprio usuário invocador, nunca pro root real; (3) `cat /proc/<PID>/uid_map` exposto o mapeamento bruto do kernel (`0 1000 1` / `1 100000 65536`) — mesma técnica de introspecção via `/proc` usada no Módulo 5 com `readlink /proc/<PID>/root`, agora aplicada a UID em vez de raiz de filesystem. Conclusão prática registrada: "root" dentro de um container Podman rootless nunca excede o privilégio que o usuário invocador já tinha no host — ao contrário do grupo `docker`, que é uma porta lateral genuína pra root real.
  - Discussão de mercado (fora da VM, conceitual): Docker domina "mindshare" por efeito de pioneiro e ecossistema entrincheirado (Docker Desktop, Docker Hub, tutoriais); Podman ganha terreno em ambientes Red Hat/RHEL, CI/CD dentro de Kubernetes (evita Docker-in-Docker) e setores regulados que exigem rootless por política. Pra vaga de DevOps, "Docker" ainda é o termo genérico universal em descrição de vaga; Kubernetes importa mais que a escolha de runtime isoladamente. Recomendação registrada: dominar Docker (expectativa universal), manter o entendimento conceitual do Podman (diferencial).
  - Dúvida paralela do Gerlan sobre migrar pra Red Hat/RHEL respondida e registrada em `~/01_Estudos/ideias-trilha-linux-server.md` (fora do git, doc de brainstorm da próxima trilha) — decisão: não migrar agora, terminar a trilha DevOps atual em Debian, e reservar RHEL-family (Rocky/Alma) como bloco específico da próxima trilha (Linux fundamentals), não motivo pra trocar a base atual.
